## Интеграция архитектурного цикла (archforge) и Compound Engineering

В проекте используются два дополняющих плагина:

- **`compound-engineering`** (EveryInc) — рабочий цикл уровня фичи: Brainstorm → Plan → Work → Review → Compound.
- **`archforge`** — архитектурный цикл: Discover → Research → Design → Decide → Document → Review.

Они чередуются, а не работают параллельно. Подробные правила — в скилле
`compound-integration` плагина `archforge`. Краткая шпаргалка ниже.

### Объединённый поток работы

```
/ce-ideate (опционально)
  -> /ce-brainstorm
       -> [задача архитектурно значима?]
            - НЕТ -> /ce-plan
            - ДА  -> /archforge:cycle (Discover -> [Research] -> Design -> Decide -> Document)
                       -> /ce-plan (на вход — ADR и brainstorm)
                            -> /ce-work
                                 -> /ce-code-review
                                      -> /archforge:review (соответствие ADR)
                                           -> /ce-compound
                                                -> [появился новый архитектурный паттерн?]
                                                     - ДА -> /archforge:adr
```

### Триггеры архитектурной значимости (запускать archforge-цикл перед /ce-plan)

- Новый модуль или каталог верхнего уровня.
- Новая зависимость в манифесте пакетов.
- Новый внешний сервис (БД, очередь, кэш, сторонний API).
- Новый протокол (введение gRPC, WebSocket, SSE и т.п.).
- Изменение модели данных, ломающее существующих потребителей.
- Изменение топологии развёртывания.
- Решение по паттерну, которого команда раньше не принимала (авторизация, модель ошибок, ретраи, кэширование).
- Пользователь явно говорит «это архитектурное решение».

### Неархитектурные триггеры (пропускаем archforge, сразу /ce-plan)

Багфиксы, новые эндпоинты или компоненты по уже сложившимся паттернам, тесты,
рефакторинг внутри границы существующего модуля, форматирование,
обновления документации.

### Владение артефактами

| Файл | Владелец |
|---|---|
| `ARCHITECTURE.md`, `STRATEGY.md` | archforge |
| `docs/architecture/{decisions,diagrams,research,reviews}/`, `decision-map.md` | archforge |
| `docs/{brainstorms,plans,solutions}/` | compound-engineering |
| `todos/` | compound-engineering |
| `AGENTS.md`, `CLAUDE.md` | общие |

Ни один файл не записывается обоими плагинами.

### Перекрёстные ссылки — обязательные

- Планы CE, затрагивающие архитектуру, **обязаны ссылаться на нужные ADR по номеру**.
- Новые ADR, появившиеся внутри задачи CE, **должны ссылаться на brainstorm/plan-документ** в секции Context.
- Compound-учения (`docs/solutions/`) **могут ссылаться на ADR**; если само учение архитектурное — заводи ADR отдельно.

### Разделение обязанностей при ревью

- `/ce-code-review` проверяет корректность, безопасность, производительность, поддерживаемость, тесты.
- `/archforge:review` проверяет соответствие ADR и архитектурные смрады (границы, связанность, слоистость).

Запускай оба на крупных PR. На небольших или неструктурных PR — только `/ce-code-review`.
