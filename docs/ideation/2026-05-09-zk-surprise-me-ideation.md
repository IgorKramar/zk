# Ideation: zk — surprise-me

Запуск `/ce-ideate` в режиме surprise-me на свежем zk-репозитории, где в ту же сессию были написаны `STRATEGY.md`, `ARCHITECTURE.md` (включая 5 открытых вопросов и 14 решений в `decision-map.md`) и `AGENTS.md`. Шесть подагентов с разными рамками рассмотрения сами выбирали, на каких поверхностях zk генерировать кандидатов; затем оркестратор отфильтровал 47 сырых + 7 кросс-рамочных кандидатов до 7 выживших.

## Контекст обоснования

**Проект.** zk — личный Rust CLI/TUI инструмент для Zettelkasten. Рабочее дерево сейчас пустое; в git-истории — прежняя реализация (UUID-ID в `metadata.rs`, YAML-frontmatter, граф на `HashMap` в памяти, поиск по тегам/заголовку/содержимому/regex/glob, делегирование редактору vim/nvim/code/emacs, работа над оформлением TUI). `STRATEGY.md` и связанные документы написаны сегодня и трактуются как привязка для всей дальнейшей работы.

**STRATEGY (дословно где можно).**
- Цель и проблема: «Заметки — plain text, который должен жить и редактироваться в той же цепочке инструментов, что и код (vim, git, grep). Существующие PKM-инструменты (Obsidian) принудительно вытаскивают граф знаний в отдельный GUI-мир.»
- Подход: «Поведение продукта диктуется доказанными паттернами Zettelkasten (Luhmann/Ahrens): atomic notes, link-before-save, фиксированная ID-схема, отказ от папок-как-таксономии и тегов-как-замены-ссылок. Research-grounded constraints держат plain-text-first модель от деградации в очередной markdown-редактор — продукт отказывается от анти-паттернов, даже если их просит пользователь.»
- Первичный пользователь: terminal-native инженеры (vim+tmux+git).
- Треки: Note graph engine · TUI & capture flow · Toolchain interop · Methodology enforcement.
- Метрики: orphan-note ratio (<5%), avg outgoing links per note (>1.5), capture latency (<5 с), cross-session usage frequency, median time-to-first-link.

**ARCHITECTURE.** Стек Rust, single binary, без runtime-зависимостей кроме `$EDITOR`/git. Формат — plain markdown, никакой проприетарной БД как первичного хранилища. Анти-паттерны: папки-как-таксономия, теги-как-замена-ссылок, Electron, cloud-sync как зависимость, переизобретение ripgrep/fzf/git, mouse-only пути в TUI. 14 открытых решений: A1–A4 (контракт на диске), B1–B3 (внутренности движка), C1–C4 (поверхность/UX), D1–D2 (interop). A1 имеет наибольший радиус влияния.

**Внешний контекст (PKM-ландшафт 2025–2026).**
- **Коллизия имени.** `zk-org/zk` (Go) — действующий CLI-Zettelkasten v0.15.4 (май 2026), 2.6k★, активный, та же ниша. SQLite-индекс, fzf/LSP-интеграция, без принуждения к методологии (сироты и глубокие папки возможны). Та же команда установки (`brew install zk` / `cargo install zk`).
- **Dendron** прекратил существование в феврале 2023, оставив пустоту в plain-markdown иерархическом PKM с CLI; их dot-notation так и не повторили.
- **Foam.** Известная проблема производительности: 65–75% CPU при 330+ заметках (Electron + полная пересборка графа в памяти, GitHub issue #347).
- **Org-roam postmortem.** SQLite быстра, но ~80% времени тратилось на преобразование данных Elisp ↔ struct; ускорение ~2,5× от обхода обёрток. Синхронизация между машинами через `org-roam-db-sync` болезненна.
- **neuron** прекращён; преемник Emanote ушёл прочь от CLI-первичности.
- **Сообщество ID-схем.** Сходимость на UUID/ULID во frontmatter + slug в имени файла; временные метки в имени отмечены как «визуально шумные»; иерархические ID Лумана — оптимизация под бумажную картотеку, теряющая смысл при полнотекстовом поиске.
- **Markdown в Rust.** `pulldown-cmark` + `pulldown-cmark-frontmatter` — идиоматичный путь; `comrak` для AST-уровня. У `pulldown-cmark` есть нативное расширение для wiki-ссылок.
- **ratatui v0.30 (2025).** Стабилен, на 30–40% меньше памяти чем Bubbletea, 60+ FPS на сложных layout. Узким местом будет архитектура приложения, не фреймворк.
- **Смежные паттерны (важные источники аналогий).** ESLint/Biome (правила warn/error, `--fix`, inline-disable), pre-commit (трёхуровневая лестница принуждения: local warn → local block → server block), Conventional Commits (соглашение как мягкая схема), TypeScript strict mode (opt-in на проект + escape-hatch), spaced-repetition системы (журнал событий вместо полных пересборок).
- **Дыра, которую никто не закрыл.** Terminal-native PKM, который одновременно (а) принуждает к ограничениям Zettelkasten на этапе записи, (б) держит постоянный индекс, (в) выставляет композируемый CLI (а не только TUI), (г) трактует нарушения методологии как линт-правила. zk-org/zk даёт (б)+(в). Org-roam даёт (б) глубоко, но привязан к Emacs. Никто не сочетает все четыре.

## Ranked Ideas

### 1. Переименовать сейчас в неконфликтующий идентификатор (например, `zkr` / `kasten` / `zett`)

**Description:** Выбрать уникальное имя до первой публикации крейта. Зарезервировать имя в cargo, GitHub-организацию и Homebrew tap. Обновить `STRATEGY.md`, `ARCHITECTURE.md`, `AGENTS.md` под новое имя.
**Warrant:** `external:` zk-org/zk (Go) — действующий проект v0.15.4 (май 2026), 2.6k★, активный, та же ниша. Каждый `cargo install zk` / `brew install zk` / поисковый запрос ведёт не туда.
**Rationale:** Решение «дверь в одну сторону», стоимость которого растёт логарифмически с возрастом проекта: дёшево сегодня (нет пользователей), мучительно через два года (битые обратные ссылки, мёртвый README, потеря поискового веса). Слабость Go-конкурента («нет принуждения к методологии») — это и есть отличительный признак zk-Rust, но никто не обнаружит этот контраст, если оба имени сталкиваются на этапе установки.
**Downsides:** Теряется имя «zk», которое автор уже мысленно присвоил (это имя каталога репозитория). Альтернатива от противного: оставить и явно позиционироваться против (по образцу Biome против ESLint, deno против node) — работает только если есть маркетинговый бюджет и терпение на поисковую раскрутку.
**Confidence:** 90%
**Complexity:** Low
**Status:** Unexplored

### 2. CLI-first архитектура; TUI — тонкий рендерер поверх CLI

**Description:** Вся бизнес-логика — в стабильных CLI-командах с JSON-выводом (`zk new`, `zk link`, `zk lint`, `zk graph --json`). TUI вызывает их; никакой логики в `tui/`. Все будущие поверхности (LSP-сервер, веб-просмотр, мобильный захват, AI-агент, vim-плагин) становятся тонкими обёртками.
**Warrant:** `direct:` `ARCHITECTURE.md` §7, анти-паттерн «не переизобретать ripgrep/fzf/git» + STRATEGY-трек «Toolchain interop». `reasoned:` git+magit, fzf+everything, ripgrep+everything получили долголетие именно потому, что отделили движок от представления.
**Rationale:** Архитектурная дисциплина почти бесплатна сейчас и дорогая позже. Закрывает решение C1 с максимальной свободой манёвра в будущем. TUI становится тестируемым через сравнение stdout с эталоном.
**Downsides:** Лёгкий налог на излишнюю общность для личного инструмента на 50 заметок. Риск разрастания CLI-поверхности, когда каждая TUI-функция обзаводится CLI-двойником.
**Confidence:** 85%
**Complexity:** Medium
**Status:** Unexplored

### 3. `zk lint` — методология как линт-правила с тремя уровнями строгости

**Description:** Движок проверки правил по образцу ESLint/clippy. Каждое правило — именованное, документированное; уровень `off|warn|error`; точечное отключение через frontmatter или HTML-комментарий; режим `--fix` где применимо. Стартовый набор `recommended-luhmann` (намеренно жёсткий по умолчанию). Правила: `zk/no-orphan`, `zk/atomic-size`, `zk/link-before-save`, `zk/no-tag-as-category`, `zk/single-h1`, `zk/stable-id`. Точки запуска: pre-commit, проверка перед сохранением, пакетный прогон в CI.
**Warrant:** `direct:` метрики `STRATEGY.md` (orphan-ratio, ссылок на заметку, time-to-first-link) — это ровно то, что измеряет линтер; архитектура уже подразумевается метриками. `external:` ESLint, clippy, biome, pre-commit, TypeScript strict mode — модель доказана в смежных областях. Внешнее исследование явно называет «lintable methodology» дырой, которую никто не закрыл.
**Rationale:** Это и есть продуктовый отличительный признак. zk-org/zk методологически нейтрален; Org-roam привязан к Emacs; Foam/Logseq не принуждают ни к чему. Каждое правило — исследовательский результат, закодированный навсегда. Сообщество добавляет свои наборы правил → корпус растёт → наборы становятся брендовыми активами (`zen-luhmann`, `ahrens-strict`).
**Downsides:** Решения по форме движка (API правил, модель уровней, `--fix`) занимают реальное время до того, как пользователь что-то увидит. Риск «паралича настроек» — слишком много правил, слишком много настройки; противоречит позиции STRATEGY «продукт отказывается от анти-паттернов», если набор по умолчанию не жёстко мнение-имеющий.
**Confidence:** 90%
**Complexity:** Medium-High
**Status:** Unexplored

### 4. Постоянный SQLite-индекс, источник событий — файловая система (без полной пересборки)

**Description:** SQLite в режиме WAL в `.zk/index.db`. Наблюдатель ФС эмиттит append-only события (`note_created`, `link_added`, `link_removed`, `note_renamed`) в журнал; SQLite — материализованная проекция. Холодный старт = воспроизведение от последнего снимка. Индекс — производный кэш, который можно удалить и пересобрать.
**Warrant:** `external:` postmortem Org-roam — SQLite быстра, но ~80% времени тратилось на преобразование Elisp ↔ struct; ускорение ~2,5× от обхода обёрток. `external:` Foam #347 — 65–75% CPU при 330+ заметках от Electron + полной пересборки в памяти. `external:` системы интервального повторения используют событийный источник для того же профиля нагрузки (много добавлений, мало изменений).
**Rationale:** Закрывает B1 с самым сильным внешним обоснованием. Задержка захвата остаётся <5 с даже на 10k заметок — индекс амортизируется. Синхронизация между машинами (тихий убийца, замеченный сообществом Org-roam) сводится к «rsync журнала» вместо «diff двух баз». Бесплатные транзакции, межпроцессные блокировки, `sqlite3` для отладки.
**Downsides:** Добавляет жизненный цикл производного артефакта (включён ли в `.gitignore`? попадает ли в `zk export`? пересобирается на `clone`?). Инварианты журнала надо тщательно поддерживать (идемпотентность при воспроизведении; миграции схемы при изменении правил). Не тривиально для v0.1 личного инструмента, если события не максимально простые.
**Confidence:** 80%
**Complexity:** Medium
**Status:** Unexplored

### 5. Зафиксировать A1 как `ULID-prefix + human-slug.md` и заморозить формат как публичный контракт

**Description:** Имена файлов вида `01J9XQK7ZB-on-fixed-ids.md`. ULID во frontmatter — канонический ID; slug в имени файла — для эргономики `ls`. Опубликовать `format-v1.md` как поставщико-нейтральную спецификацию; набор тестов в CI валит любой PR, ломающий совместимость парсера. Изменение формата — явный `format-v2` плюс инструмент миграции.
**Warrant:** `direct:` `decision-map.md`: A1 блокирует A2/A3/A4/B1/B2/C4/D1 — наибольший радиус влияния среди 14 решений. `external:` сообщество Zettelkasten сошлось на UUID/ULID во frontmatter + slug в имени файла; временные метки в имени отмечены как «визуально шумные»; иерархические ID Лумана — оптимизация под коробку с карточками, бесполезная при полнотекстовом поиске.
**Rationale:** Все остальные решения дешевеют, потому что первичный ключ стабилен. Замороженный формат означает: ноль миграционного инструментария когда-либо; индексы обратных ссылок только пополняются; пользовательские скрипты работают через 10 лет. Развязка жизненного цикла кода и данных — это сама суть рычага.
**Downsides:** Слишком ранняя заморозка может закрепить ошибку (поведение ULID при смене таймзоны, нормализация slug). Смягчение: явные минорные версии `format-v1` для добавочных изменений; набор тестов ловит поломки; fuzz-корпус граничных случаев имён файлов.
**Confidence:** 85%
**Complexity:** Low (выбор) → Medium (дисциплина поддержки спецификации)
**Status:** Unexplored

### 6. Mise-en-place: разделить `capture` (режим подачи, <5 с) и `prep` (методичный, строгий)

**Description:** Два потока. `zk capture` (или одно нажатие в TUI) — минимальное трение, дозапись в `inbox.md`-буфер с заглушкой ID, без обязательной ссылки, без проверок. `zk prep` (или отдельная вкладка TUI) — методичная обработка вчерашнего входящего: присвоить финальный ID, разбить на атомы, связать с существующими, переозаглавить, продвинуть в граф. Ограничения (`zk/link-before-save` и пр.) применяются на этапе `prep`, а не `capture`. Заметки во входящем считаются сиротами, но в метрике orphan-ratio считаются отдельно.
**Warrant:** `reasoned:` ресторанные кухни решили эту коллизию век назад — подача (service) ограничена задержкой; заготовка (prep) ограничена методикой; смешивать — губить обе. `direct:` метрики `STRATEGY.md` в напряжении: capture latency <5 с и orphan-ratio <5% тянут в разные стороны, если применять к одному моменту.
**Rationale:** Разрешает центральную коллизию «методичность vs поток» на уровне архитектуры, а не на уровне правил. Обе метрики выигрывают. Определяет ежедневный ритуал, который пользователь реально совершает — привычка строится вокруг методики, а не против неё. Хорошо ложится на пункт 3 (`zk lint`) — линтер по умолчанию проверяет только заметки, прошедшие `prep`.
**Downsides:** Добавляет режим, который пользователь обязан принять; если `prep` пропускается — `inbox.md` становится write-only кладбищем (это и есть текущая болезнь Obsidian Daily Notes; в сообществах вроде Forte PARA большинство пропускает шаг продвижения). Смягчение: фоновой процесс `zk prep` напоминает или показывает счётчик.
**Confidence:** 70%
**Complexity:** Medium
**Status:** Unexplored

### 7. Запечь синтетический корпус 5 000 заметок в CI как принуждающую функцию для B1/B3/D1

**Description:** Сгенерировать синтетический набор заметок объёма 5k (реалистичная плотность ссылок, frontmatter, разные размеры) на этапе сборки. CI допускает альтернативу B1 только если p95 холодного старта и p95 capture latency на этом масштабе укладываются в бюджет. Дополнительный 50k stress-уровень — небло́кирующий.
**Warrant:** `direct:` внешнее исследование — Foam #347 документирует обвал производительности на ~330 заметках (полная пересборка в памяти + Electron). `direct:` git-история — прежняя реализация zk использовала граф на `HashMap` в памяти (тот же анти-паттерн). Без бенчмарка-ворот новая реализация унаследует тот же выбор и тот же провал.
**Rationale:** Дешёвая конкретная принуждающая функция. Превращает B1 и смежные решения из теоретического спора в эмпирический. Ловит регрессии производительности рано. Становится частью идентичности проекта («мы бенчмарчим на 5k с первого дня»).
**Downsides:** Генератор корпуса — реальная работа (1–2 дня). 5k может оказаться маловато для активного пользователя на третий год; повышение базовой планки до 50k увеличивает время CI. Синтетический набор может не воспроизвести патологические реальные паттерны.
**Confidence:** 90%
**Complexity:** Low
**Status:** Unexplored

## Кросс-рамочные сходимости

Эти темы независимо подняты ≥3 рамками — сильный сигнал того, что они не случайны:

- **X1. Engine kernel + тонкие UI-клиенты** (LSP / демон / плагин). Поднята в рамках 2/3/4/6 в шести разных формулировках. Сходимость: бизнес-логика не в TUI; движок выставляет стабильный интерфейс; UI — тонкие клиенты. Канонический выживший — пункт 2.
- **X2. Methodology-as-lints как сам продукт.** Поднята во всех шести рамках. Сходимость: форма ESLint, применённая к Zettelkasten; линт-движок и есть то, что никто не выпустил. Канонический выживший — пункт 3.
- **X3. Коллизия имени с zk-org/zk** — четыре разных разрешения: переименовать сейчас (дешёвая страховка), оставить и позиционироваться против, плагин Obsidian делает коллизию неважной, единая спецификация формата делает имя вторичным. Канонический выживший — пункт 1; альтернатива от противного отмечена в его минусах.
- **X4. Постоянный индекс через источник событий ФС** (vs полная пересборка в памяти). Поднята в рамках 2/4/5/6. Сходимость: append-only журнал событий + проекция SQLite; синхронизация между машинами становится merge журнала. Канонический выживший — пункт 4.
- **X5. Совместимость с vault Obsidian как рычаг распространения.** Поднята в рамках 3/4/6. Сходимость: не бороться с гравитацией Obsidian; обходить через совместимость формата. Не выделена в отдельного выжившего; частично закрывается пунктом 5 (открытая спецификация формата) и упомянута в минусах пункта 1.
- **X6. Capture vs prep как раздельные режимы** (разрешает напряжение метрик). Поднята в рамках 2 и 5 (mise-en-place). Канонический выживший — пункт 6.
