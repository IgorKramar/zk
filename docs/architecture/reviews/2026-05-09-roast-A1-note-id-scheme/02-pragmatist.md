# Pragmatist findings: A1 — Note ID scheme + filename layout (ULID + slug)

- **Target**: `docs/architecture/research/2026-05-09-A1-decision-summary.md`
- **Date**: 2026-05-09
- **Role**: pragmatist (operational realism)

## Summary

Для соло-разработчика, который ведёт собственные заметки на собственном ноутбуке и синхронизирует их через git — это **операционно близко к минимальному жизнеспособному уровню**. Через месяц всё работает; через шесть месяцев накопится первая партия рукотворных edge cases (CJK-заметка с пустым slug, ручной edit ULID в frontmatter, sync-конфликт по rename); через два года настоящая стоимость — это не сами пять крейтов, а **`gray_matter` write-back fidelity** и **отсутствие миграционного инструмента, когда выяснится, что format-v1 нужно править**. Есть два неназванных операционных долга, которые в декларации спрятаны за словами «достаточно», «приемлемо», «O(1)».

## Findings

### P-1: Пять крейтов выглядят дёшево, но `gray_matter` — единственная нетривиальная supply-chain ставка

**Категория**: hidden overhead / supply chain over 5 years

**Реальность**: `ulid` 1.2.1 — зрелый, активный, ~16M downloads, рисков мало; `slug` + `deunicode` — практически замороженные, low-churn, рисков мало; `serde_yaml` — **уже архивирован автором (David Tolnay) в марте 2024**, помечен «no longer maintained» в crates.io. Это значит, что через 5 лет вы будете либо мигрировать на `serde_yml` / `saphyr` / `yaml-rust2`, либо нести deprecated-зависимость. `atomic-write-file` — узкоспециализированный, mid-popularity (low downloads); если автор забросит — переписать на ~50 строк своего кода реалистично. `gray_matter` — самый «живой», но maintained двумя людьми; lossy write-back (зафиксировано в research §4) — это **гарантированный будущий day-of-pain**: первый раз, когда пользователь вручную добавит комментарий в frontmatter и zetto его сотрёт после retitle, — это будет неприятный bug-report на самого себя.

**Cost estimate**: миграция `serde_yaml` → замена при первом security advisory или MSRV-конфликте — ~4–8 часов (одна точка вызова в проекте, тривиальная замена). Подмена `gray_matter` на YAML-AST-парсер при потере fidelity — **10–20 часов**, если откладывать; **2–3 часа**, если делать вместе с сменой `serde_yaml`. Совокупная maintenance-нагрузка пяти крейтов — ориентировочно 0–4 часа/квартал в спокойный год, 8–16 часов/квартал в год breaking-change'а одного из upstreams.

**Severity**: medium. Не выбьет проект из колеи, но именно через эти крейты пройдёт большинство будущей operational боли — и `serde_yaml` уже на пути к deprecation сегодня, не «через 2 года».

### P-2: «`std::fs::rename` достаточна» — это самый недосказанный фрагмент решения

**Категория**: hidden overhead / deployment risk (для personal use = «sync risk»)

**Реальность**: декларация утверждает atomicity на POSIX и Windows ≥10 1607. Это правда для одного процесса на одной машине. Но **главный сценарий zetto — git-sync между ноутбуками**, и rename там перестаёт быть атомарным с точки зрения пользователя: на машине A retitle переименовал файл, на машине B параллельно произошёл edit того же файла под старым именем — git превратит это в delete-old + add-new + modify-old, и merge-конфликт окажется не на содержимом, а на самой identity файла. Восстановление — ручное (cherry-pick + ручной rename + ручной rebase). У одного пользователя это случится не часто (нельзя одновременно быть на двух машинах) — но **случится** по сценарию «отредактировал на ноутбуке, не запушил, продолжил на десктопе из последнего синка». Также: `std::fs::rename` overwrites target — research §5 это упомянул как hazard, summary опустил. ULID-collision действительно vanishingly unlikely, но **slug-collision при manual edit пользователя** (он переименовал ULID в frontmatter, не сделав того же в filename) теоретически возможен, и `std::fs::rename` молча перезапишет.

**Cost estimate**: **~1 ручное вмешательство в год** при двухмашинном использовании; **~30 минут** на восстановление каждый раз (вспомнить, что произошло, разрулить git). За 5 лет — несколько часов суммарно. Не критично, но нужно один раз пройти и записать runbook самому себе («что делать если sync дал rename-conflict»). Цена забытого runbook — час паники в первый раз.

**Severity**: low–medium. Низкая частота, средний impact когда случится. Главное — это не в документе.

### P-3: Backlinks по `[[ULID]]` через filename-glob — O(1) на один lookup, но **O(N) на полный directory scan каждый раз**

**Категория**: hidden overhead / day-1 vs steady-state

**Реальность**: декларация спрятала индексный долг во фразу «нет необходимости в persistent index для одного lookup; индекс понадобится для inverse (backlinks), но это решается отдельно (B1)». Что это значит operationally: на каждое разрешение ссылки `[[01J9XQK7...]]` zetto делает либо `readdir` каталога заметок (на 100 заметках — копейки, на 5000 — заметные миллисекунды на холодном FS-кеше), либо `glob` с walking. Бюджет index lookup из ARCHITECTURE §2.1 — 100 ms; на 1k+ заметок без индекса этот бюджет реально начнёт скрипеть, и B1 из «отдельного ADR» превратится в **блокер для capture latency**. Декларация это упоминает — но как «решается отдельно». Operationally это значит: **между моментом «A1 в проде» и моментом «B1 решён» ваш реальный capture-flow живёт на linear scan**. Если вы доползёте до 2k заметок до B1 — это будет первая постоянная боль продукта, ровно по той метрике, которую STRATEGY называет ведущей.

**Cost estimate**: цена откладывания B1 = **0 при ≤500 заметок, ощутимо при 1k+, неприемлемо при 3k+**. Сколько потребуется в `B1` потом: persistent index — 1–2 недели работы (sqlite или sled, watcher, invalidation). Цена *неустранения* — постепенная деградация capture-latency, замаскированная FS-кешем; вы заметите её слишком поздно.

**Severity**: medium. Не сегодня, не через месяц, но это первый «iceberg under the proposed waterline» — ровно в смысле, который этот шаблон роли просит назвать.

### P-4: Format-v1 контракт зафиксирован, миграционного инструмента нет

**Категория**: deployment risk / on-call

**Реальность**: §«Контракт публичного формата» гласит: любое изменение filename layout требует format-v2 + `zetto migrate format-v1 format-v2`. Этот инструмент не существует, его нет в roadmap, и оценки на его реализацию нет. Это **жёсткое будущее обязательство, замаскированное под нейтральную формулировку**. Реальный сценарий: через 6–12 месяцев вы захотите поменять что-то невинное (например, добавить опциональный config-knob для CJK fallback на ULID-only filename — он уже маячит как open question). Если это меняет filename — нужен migrate-tool. Если migrate-tool ещё не написан — выбор: либо ломать собственные заметки, либо откладывать улучшение, либо писать migrate-tool под давлением. Это классическая ловушка «мы сделаем это правильно потом».

**Cost estimate**: первая реализация `zetto migrate` — **8–16 часов** (read all .md, parse frontmatter, derive new layout, atomic-rename, dry-run mode, rollback). Цена «в нагрузку, под давлением» — выше: 16–24 часа, потому что внезапно нужны тесты, dry-run, rollback на собственный vault. Дешевле сделать заранее, ещё дешевле — **записать в декларацию trigger**: «когда vault достигнет N заметок, написать migrate; до тех пор любая идея смены layout откладывается».

**Severity**: medium. Не сейчас, но именно эта статья контракта — то, что превращает A1 из «решения» в «обязательство на годы».

### P-5: CJK fallback — заявлен как open question, но в operationally-значимом месте

**Категория**: tooling / day-1 ergonomics

**Реальность**: §«Slug normalization» открыто признаёт: японские кандзи мапятся на путунхуа-pronunciation, что нечитаемо для японского читателя; для японско-тяжёлых vault-ов — open question, fallback на ULID-only filename. Operationally для конкретно этого пользователя (русско-английский primary, по STRATEGY и ARCHITECTURE) это **скорее всего никогда не активируется**. Но если активируется — то в момент создания первой такой заметки, и поведение будет: `<ULID>.md` без slug-а вообще. Это **inconsistent UX**: большинство заметок читаемы в `ls`, одна стенка ID. Не баг, но мелкая шероховатость, о которой пользователь не будет помнить через 6 месяцев («почему эта заметка без slug?»). Проще принять решение сразу: либо fallback на транслит latin-character-by-character (`猫` → `cat-something-mei`), либо явно фиксировать «в v1 кандзи дают ULID-only — known limitation, см. ADR-0002».

**Cost estimate**: 0 если не возникнет; ~30 минут на дебаг-«почему у меня тут странный filename» когда возникнет первый раз. Решение — одна строка в ADR-0002 («known limitation»).

**Severity**: low. Реалистичная для соло-русско-англо-говорящего пользователя.

### P-6: Локальная разработка и фидбек-цикл — operationally здоровы

**Категория**: tooling & ergonomics (positive finding)

**Реальность**: пять крейтов, все pure Rust, никаких системных зависимостей, никакой БД, никакого daemon. `cargo test` с in-process FS (через `tempfile`) даст быстрый feedback loop. Никакого Docker, никакого «поднять половину стека для итерации». Это **именно то, что декларация подразумевает, но не говорит явно**, и это сильная сторона. Test fixtures — несколько папок с .md файлами в `tests/fixtures/`. Capture-flow можно тестировать end-to-end в <100 ms. Decision действительно совместим с принципом «inner loop остаётся быстрым», который для соло-проекта определяет, заброшен ли проект через год или нет.

**Cost estimate**: **отрицательный** — этот выбор экономит 5–10 часов операционной возни, которые потребовал бы вариант с persistent SQLite-индексом или Tantivy с самого начала. То есть A1 как сформулирован сохраняет соло-темп.

**Severity**: positive (отметка). Этот пункт — баланс к P-1…P-5.

## What's understated in the proposal

- «`std::fs::rename` достаточна для slug-rename» — достаточна для атомарности FS-операции. Не достаточна для sync-семантики через git между двумя машинами, и не достаточна для случая overwrite-target.
- «индекс понадобится для inverse (backlinks), но это решается отдельно (B1)» — переводится как «между сегодня и B1 ваш граф работает на linear scan». Слово «отдельно» не убирает зависимость; оно её перекладывает.
- «Migration tool (`zetto migrate format-v1 format-v2`)» — упомянут как if-clause, но ни срок, ни owner, ни trigger не зафиксированы.
- «latest» в колонке версий для `slug`, `gray_matter`, `serde_yaml`, `atomic-write-file` — это понятно для дизайна, но в `Cargo.toml` это будет конкретная пиннутая версия. Декларация не фиксирует policy («pin to ~minor», «follow latest», «cargo-audit еженедельно»).
- «Capture latency budget на этом шаге < 50 ms» — это правда для warm cache. Холодный FS write первого файла дня на крутящемся диске — десятки ms. «<50 ms» как абсолют — ложная точность.

## What's missing entirely

- **Backup/restore-стратегия для самого vault.** Git синхронизирует, но не бэкапит. Один абзац-disclaimer был бы честен.
- **`Cargo.lock` policy.** Соло-проект, бинарь — `Cargo.lock` коммитится. Зафиксировать рядом с пятью крейтами.
- **`cargo audit` / `cargo deny` ритуал.** Раз в квартал прогнать `cargo audit` — 5 минут.
- **Что zetto делает, если frontmatter отсутствует / `id:` поле отсутствует.** Декларация требует `id:` обязательным, но не описывает поведение на «грязных» .md.
- **Runbook для самого себя.** Краткий «troubleshooting» в README или в `docs/architecture/runbook.md`.
- **Версии MSRV.** Зафиксировать `rust-version` в `Cargo.toml` — 5 минут.

## What's actually realistic

Для одного пользователя, одной машины-как-primary плюс одного-двух sync-эндпоинтов через git — A1 как сформулирован реалистичен **с двумя поправками к декларации**, которые ничего не меняют в дизайне, но фиксируют операционную реальность:

1. Заменить «latest» на конкретные пиннутые версии в `Cargo.toml` сразу, и зафиксировать в одной строке ADR-0002: «`Cargo.lock` коммитится; `cargo update` — раз в квартал с `cargo audit` перед коммитом; `serde_yaml` помечен deprecated upstream — мигрировать к `saphyr`/`serde_yml` при следующем breaking event».
2. Добавить в раздел «Контракт публичного формата» один параграф: trigger для `zetto migrate format-v1 → format-v2`. Например: «Migration tool пишется тогда (и только тогда) когда первая идея, требующая format-v2, проходит фазу Decide».

Всё остальное — `gray_matter` write-back fidelity, B1-индекс, CJK fallback, sync-rename — это известные ограничения, которые **жить с ними дешевле, чем устранять заранее**, при условии, что они **названы вслух один раз** в ADR-0002 как known limitations с явным trigger-ом для пересмотра.
