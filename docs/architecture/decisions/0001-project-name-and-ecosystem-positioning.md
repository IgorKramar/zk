# ADR-0001: Rename project from `zk` to `zetto`

- **Date**: 2026-05-09
- **Status**: Accepted
- **Authors**: Igor Kramar

## Context

Текущее имя `zk` унаследовано от прежней реализации (последний реальный коммит `293a1e4`, ноябрь 2024) и было выбрано до того, как написана `STRATEGY.md`. После того как 2026-05-09 `STRATEGY.md`, `ARCHITECTURE.md` и `docs/architecture/decision-map.md` зафиксировали целевое позиционирование (research-grounded constraints, lintable methodology), коллизия имени стала видна как load-bearing проблема:

- **Внешняя:** [zk-org/zk](https://github.com/zk-org/zk) — действующий Go-CLI для Zettelkasten v0.15.4 (май 2026), 2,6k★, активно поддерживается, та же ниша. Та же команда установки (`brew install zk` / `cargo install zk`), те же поисковые запросы, те же шpargalки. SEO/install-collision вытаскивает пользователя не на тот продукт каждый день.
- **Внутренняя:** найдено `/archforge:observe` 2026-05-09 как **O-1** (high severity) — «дверь в одну сторону, стоимость которой растёт логарифмически с возрастом проекта». Подтверждено `/ce-ideate` 2026-05-09 как **Idea #1** (выживший, уверенность 90%, complexity Low).

Перед принятием решения проведена внешняя проверка имени-кандидата `zetto` по ключевым реестрам и каналам, важным для Rust-CLI:

| Канал | Состояние |
|---|---|
| crates.io | свободно (0 совпадений) |
| Homebrew core | свободно (проверено через `formulae.brew.sh/api/formula.json`) |
| AUR (Arch) | свободно |
| PyPI | свободно (на случай Python-обвязки) |
| npm `zetto` | занято заброшенным 4-летним Vue/TS CRUD-виджетом (автор `mitto98`); другая ниша |
| npm `@zetto/*` MCP-namespace | занят активным брендом `zetto-memory` / `@zetto/mcp-server` (shared memory pool для AI-агентов через MCP); смежная, но другая область |
| GitHub username `zetto` | занят неактивным аккаунтом Tomáš Škarecký (2 старых C#-репо); некритично, репозиторий живёт под `IgorKramar/zetto` |
| GitHub-репозитории `zetto` в нише PKM/notes | поиск даёт 94 совпадения; **ни одно не относится к Zettelkasten/PKM** |
| Семантика | производное от `Zettel` (нем. «карточка»); 5 букв, 2 слога; читается во всех целевых языках |
| Японская фонетика | ゼット (zetto) = «буква Z»; известный поп-культурный пример: Dragon Ball Z (ドラゴンボールZ, «Доragonbōru Zetto») — пригодно как визуальный якорь для иконки приложения |

Коротко: для Rust-CLI как целевого канала имя `zetto` чистое в каждом критичном реестре; коллизии лежат в смежных областях (npm-MCP-namespace, GitHub-username), которые можно либо обойти (scoped-namespace), либо принять как ограничение.

## Decision

Переименовываем проект `zk` → `zetto`.

- Имя крейта на crates.io: `zetto`. Резервируем при первой возможности (squat-defense `v0.0.0-stub` допустим до v0.1).
- Имя бинарника: `zetto`, плюс `z` как short-form symlink в установочной инструкции — стоимость набора на одну букву меньше, CLI-минимализм терминальной аудитории сохранён.
- Репозиторий: `github.com/IgorKramar/zetto` (переименование `IgorKramar/zk` создаёт автоматический redirect старых URL).
- Конфигурация: `~/.zetto/notes/` и `.zettorc` (переход с `~/.zk/` и `.zkrc` через команду `zetto migrate`).
- Описание для GitHub и crates.io начинается со слова «Zettelkasten» — снимает путаницу одновременно с `zk-org/zk`, npm-`zetto-memory` и общим словом «zk» (zero-knowledge).

## Consequences

### Easier

- **Discoverability.** Запросы «zetto zettelkasten», «zetto cli», «zetto rust», `cargo install zetto`, `brew install zetto` ведут только в один продукт. Каждая публикация (блог-пост, README, Stack-Overflow-ответ) попадает в собственный SEO-силос.
- **Дифференциация от `zk-org/zk`** видна на этапе установки и первого знакомства, без необходимости в пояснительной прозе.
- **Брендовая идентичность.** `zetto` — короткое произносимое слово с собственным семантическим ядром (от `Zettel`). Японская ゼット = «буква Z» — пригодный визуальный якорь для иконки (буква «Z» как фирменная форма).
- **Crate-namespace-strategy.** Один claim на crates.io (`zetto`); никаких суффиксов вроде `zettelkasten-cli`.

### Harder

- **Стоимость набора.** 5 символов вместо 2. Митигируется symlink-ом `z` (одна буква, на одну меньше чем `zk`) — упоминается в установочной инструкции; пользователь может сделать собственный shell-alias.
- **Миграция конфигурации существующих установок.** В текущей фазе вероятность ≈ 0 (проект pre-alpha, рабочее дерево пустое), но при первой публикации потребуется команда `zetto migrate`, обнаруживающая `~/.zk/` и предлагающая перенос.
- **Синхронные правки документации.** `STRATEGY.md`, `ARCHITECTURE.md`, оба `README.md` / `README.ru.md`, `decision-map.md` обновляются сразу следующим коммитом после акцепта этого ADR (см. Implementation).
- **Поисковая дисциплина первых месяцев.** При коммуникации проекта стоит явно писать «zetto (Zettelkasten)» в социальных сетях и на HN/Reddit, чтобы алгоритмы быстрее ассоциировали имя с нишей.

### Risks accepted

- **`@zetto/*` MCP-namespace на npm — out of scope.** Активный бренд `zetto-memory` / `@zetto/mcp-server` существует в смежной области (shared memory для AI-агентов через MCP). Если zetto в будущем выпускает собственный MCP-сервер (вероятно в свете AGENTS.md-интеграции с Claude-плагинами), он живёт под scoped-именем вроде `@kramar/zetto-mcp`, а не `@zetto/*`. Это документированное ограничение, не уступка — мы не пытаемся занять чужой неймспейс.
- **GitHub username `zetto` — недоступен.** Принимаем, что репозиторий живёт под `IgorKramar/zetto`; вариант `github.com/zetto/zetto` для нас закрыт. Не блокирует ни один канал распространения.
- **Один заброшенный npm-пакет `zetto`** (Vue/TS CRUD). Если когда-нибудь появится npm-обвязка, она будет под scoped-именем (`@kramar/zetto-cli`) или suffix-вариантом (`zetto-cli`).
- **Долгосрочный SEO-overlap с `zetto-memory`.** Если этот бренд вырастет, в выдаче возможна частичная конкуренция. Митигация: ассоциация «zetto = Zettelkasten» в первой строке каждого описания с первого дня; ниши достаточно различны (CLI для людей vs MCP-сервер для агентов).

## Alternatives considered

### 1. Сохранить `zk`, позиционироваться против `zk-org/zk`

«Их zk не принуждает; наш zk принуждает.» По образцу Biome против ESLint+Prettier, deno против node — коллизия как бесплатная сравнительная SEO. *Сильная сторона:* нулевая стоимость миграции, преемственность с прежним каталогом. *Слабая сторона:* SEO/install-коллизия — это не одноразовая проблема, а ежедневная (каждый блог-пост, каждая установка, каждое предложение коллеге «попробуй zk» требует уточнения). Маркетинговый бюджет личного OSS-проекта — нулевой; SEO-терпение измеряется годами; стоимость растёт логарифмически с возрастом — дёшево сегодня, брутально через два года. *Lost because:* не подкреплено бюджетом и временем, которые делают такое позиционирование жизнеспособным.

### 2. Suffix-deconfliction: `zk-rs`, `zkr`

*Сильная сторона:* сохраняет `zk`-ментальную модель; минимальный когнитивный сдвиг. *Слабая сторона:* подчиняет идентичность Go-проекту (читается как «форк» или «Rust-порт», что неправда — это семантически отдельный продукт с другой методологической позицией). Команда `zk-rs new` неуклюжа; в устной коммуникации звучит как «зэт-кэй-эр-эс». *Lost because:* фиксирует zk-org/zk как референсный продукт, от которого мы отстраиваемся, вместо того чтобы строить собственный.

### 3. Альтернативные немецкоязычные корни: `kasten`, `zett`, `noteforge`

*Сильная сторона:* сохраняют связь с Zettelkasten-методологией; некоторые могут быть свободны. *Слабая сторона:* для них не проведена та же due-diligence-проверка по реестрам. `kasten` пересекается с немецкими SaaS-/storage-проектами; `zett` слишком короткое (4 буквы — конкуренция за namespace высока); `noteforge` теряет немецкоязычную семантическую связку. *Lost because:* для каждого варианта потребовался бы отдельный исследовательский раунд; для `zetto` он уже проведён. Возврат к этой ветке — открытие закрытого вопроса.

### 4. Status quo: ничего не делать

Не переименовывать; терпеть коллизию или откладывать решение до v0.1. *Сильная сторона:* нулевые усилия немедленно. *Слабая сторона:* стоимость коллизии на каждой инсталляции и каждом поиске; решение всё равно придётся принять до v0.1; откладывание превращает обратимое сегодня в дорогое завтра. *Lost because:* отложенная одна-направленная-дверь — это не сохранение опции, а удорожание неизбежного.

- **`docs/architecture/decision-map.md`** — D3 переходит open → decided со ссылкой на этот ADR; уровень 0 Suggested order пересобирается (D3 уходит из «next up»). Уже сделано этим коммитом.
- **`docs/architecture/README.md`** — ADR index получает запись. Уже сделано этим коммитом.
- **`ARCHITECTURE.md` §5 Decision Index** — получает запись. Уже сделано этим коммитом.
- **`AGENTS.md`** — без правок (интеграционный блок не упоминает имя проекта напрямую).

Намеренно НЕ правится:

- **`docs/ideation/2026-05-09-zk-surprise-me-ideation.md`** — оставлен как исторический snapshot до решения. Переписать его задним числом — потерять трассируемость «как мы пришли к этому решению».
- **Git-история до `293a1e4`** — часть исторического следа, не переименовывается.

Вне репозитория (отдельный пользовательский шаг):

- **nvim-плагин-namespace** `zetto.nvim` — занять при подходе к C5 (Editor-integration strategy).
